# 结果融合层 — 产品设计文档

## 1. 产品视角：用户不关心"走哪条链路"

从两套测试集的真实问题看，用户在财政场景下问的问题天然是混合的：

| 用户真实意图 | 例子 | 需要什么 |
|-------------|------|---------|
| 查一个数 | "2025年省本级卫生健康支出的执行金额是多少" | 精确数字（text2sql） |
| 查数+理解 | "2019年全省一般公共预算收入总计多少，由哪几部分构成" | 数字 + 解释（两条链路） |
| 对比分析 | "2025年和2024年新增地方专项债项目有哪些变化" | 数据对比 + 政策背景 |
| 理解概念 | "一般公共预算收入的口径是什么" | 知识解释（RAG） |
| 政策解读 | "2024年预算安排中是怎么加大科技投入的" | 政策文档 + 可能的数据佐证 |
| 多轮追问 | "比上年高多少个百分点" | 上下文补全 + 数据计算 |

**用户不区分"这是问数还是问答"——他们只想得到完整、准确的回答。** 预判式的路由（先判断再执行）在边界模糊时必然出错。

## 2. 核心设计：始终并行，事后融合

```
用户问题
    │
    ▼
统一意图提取（extract → UnifiedIntentDict）
    │
    ┌───────────────────┴───────────────────┐
    ▼                                       ▼
Text2SQL 链路                            RAG 链路
- 如果槽位不全 → 返回"待补全"             - 始终执行检索
- 如果槽位齐全 → 执行 SQL                   - 用 RagIntent.need_* 调整检索策略
    │                                       │
    │  SqlExecutionResult                    │  RagResult
    └───────────────────┬───────────────────┘
                        ▼
              结果融合层（Fusion）
              - 对齐（时间、口径、实体）
              - 数值冲突检测
              - 可信度评分
              - 综合生成回答
                        │
                        ▼
                   最终回答
```

### 关键变化

| 当前做法 | 改为 |
|---------|------|
| dispatch 决定跑不跑 RAG | **RAG 始终执行**（need_* 只影响检索策略，不影响跑不跑） |
| dispatch 决定跑不跑 text2sql | text2sql 只在槽位齐全时执行，缺失时返回"待补全" |
| 两条链路结果各自返回 | **融合层统一处理**——对齐、去重、冲突检测、可信度评分 |

## 3. 融合层的具体行为

### 3.1 场景一：纯查数（text2sql 有结果，RAG 无相关知识）

```
问题: "2025年省本级卫生健康支出的执行金额是多少"
text2sql → 42.3亿元
RAG     → 检索到相关预算解读文档片段
融合    → 回答："42.3亿元。（数据来源：预算执行表）根据XX年预算解读，卫生健康支出主要用于..."
```

### 3.2 场景二：纯知识（text2sql 无槽位，RAG 直接回答）

```
问题: "河北省的减税降费政策有哪些"
text2sql → 槽位不全（缺少科目/指标）
RAG     → 检索到政策文档，生成回答
融合    → RAG 结果直接输出，附带文档引用
```

### 3.3 场景三：查数+解释（两边都有结果）

```
问题: "2019年全省一般公共预算收入总计多少，由哪几部分构成"
text2sql → 3742.5亿元，{税收: 2800亿, 非税: 942亿}
RAG     → 检索到2019年预算解读中关于收入构成的说明
融合    → 先给数字+表格，再给政策解读，
          交叉验证：RAG文档中的数字与SQL结果是否一致？
          不一致时标注可信度差异
```

### 3.4 场景四：两边都没结果

```
text2sql → 执行失败/空结果
RAG     → 未找到相关信息
融合    → 明确告知用户："当前数据中未找到相关信息。建议补充XX条件后重试。"
```

## 4. 融合规则

### 4.1 数值对齐

当 text2sql 和 RAG 都给出同一指标的数值时：
- **口径对齐**：确认两边的统计数据口径一致（如都是"全省""一般公共预算""收入"）
- **单位对齐**：统一为亿元
- **时效对齐**：确认两边数据对应同一时间范围

### 4.2 冲突处理

| 冲突类型 | 处理策略 |
|---------|---------|
| 数值不一致（微小差异） | 两者都展示，标注差异来源 |
| 数值不一致（重大差异 >10%） | 优先采信 text2sql（数据库是权威来源），标注 RAG 文档可能版本过旧 |
| 语义冲突（RAG说增收，SQL算出来是减收） | 以 SQL 计算为准，RAG 作参考说明 |
| 时效冲突（RAG文档年份 ≠ 查询年份） | 标注 RAG 文档时效，优先使用同年度文档 |

### 4.3 可信度评分

| 来源 | 基础可信度 | 加权因素 |
|------|-----------|---------|
| text2sql 精确结果 | 0.95 | 数据质量、执行成功、槽位完整度 |
| RAG 最新年份文档 | 0.85 | 文档年份匹配、地区匹配 |
| RAG 不同年份文档 | 0.70 | 年份差异越大，可信度越低 |
| RAG 无匹配文档 | 0.0 | 不采用 |

## 5. 回答生成优先级

融合层决定最终回答的组成和顺序：

1. **text2sql 结构化结果**（如果有）— 数字、表格、图表
2. **RAG 政策解释**（如果有）— 背景说明、构成分析、口径定义
3. **交叉验证标注**（如果两边都有）— 数据一致性说明
4. **引用来源**（必须）— text2sql 标注数据表/字段，RAG 标注文档名
5. **槽位补全提示**（如果 text2sql 槽位不全）— 告知用户还缺什么条件

## 6. 与当前架构的对接

当前已有：
- `UnifiedIntentExtractor` → `UnifiedIntentDict`（text2sql 17字段 + rag 5字段）
- `ConcurrentTaskDispatcher` → `DispatchPlan`
- RagAgent / RagService / HybridRetriever

需要改造：

| 模块 | 变化 |
|------|------|
| `dispatch_execution.py` | text2sql 保持按槽位决定；**RAG 始终执行** |
| `result_fusion_analysis/` | 从骨架→真实实现（对齐、冲突、评分、合成） |
| `orchestrators/assistant_request_orchestrator.py` | 从骨架→编排融合流程 |
| `RagIntent` | 不变——need_* 继续指导检索策略，但不决定跑不跑 |

## 7. AIAssistant 框架落点

```
extract() → UnifiedIntentDict
    │
    ├──→ text2sql链路（槽位齐全→执行SQL；不全→返回待补全）
    │
    ├──→ RAG链路（始终执行，need_*调整策略）
    │
    └──→ result_fusion_analysis/
         ├── alignment.py        — 时间/口径/实体对齐
         ├── conflict_resolution.py — 数值冲突检测
         ├── confidence_scoring.py  — 可信度评分
         └── answer_synthesis.py    — 综合生成最终回答
```

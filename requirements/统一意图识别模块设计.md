# 统一意图识别模块 — 输入输出设计

## 1. 模块定位

在 AIAssistant 框架的请求生命周期中，本模块位于 `request_preprocessing` 和 `dispatch_execution` 之间：

```
problem_intake → request_preprocessing → ★ 统一意图识别 ★ → dispatch_execution
                                                    │
                                              ┌─────────┴──────────┐
                                              ▼                    ▼
                                        Text2SQL链路          RAG链路
```

**职责**：从用户自然语言问题中一次性提取所有财政语义，输出一个结构化字典。字典包含两套子集——text2sql 子集和 rag 子集。下游链路各取所需，不再各自重复解析。

**核心原则**：一次提取，两路消费。LLM 优先，规则兜底。不设公共中间层。两条链路独立从 LLM 原始输出构造各自的子集，互不依赖。

---

## 2. 模块输入

| 输入项 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| `question` | `str` | 是 | 当前轮用户问题原文 |
| `history_messages` | `List[dict]` | 否 | 最近 N 轮历史消息，用于指代消解和上下文补全 |
| `region_code` | `str` | 否 | 用户地区权限码（9位行政区划编码），用于地区过滤 |

```python
# 输入示例
{
    "question": "2019年全省一般公共预算收入总计多少，由哪几部分构成",
    "history_messages": [...],
    "region_code": "130000000"
}
```

## 3. 模块输出 — UnifiedIntentDict

### 3.1 整体结构

```python
UnifiedIntentDict = {
    "text2sql":  { ... },   # Text2SQL 专用子集 — 17个字段
    "rag":       { ... },   # RAG 专用子集 — 5个字段
}
```

### 3.2 路由推导

**路由不是独立步骤，而是字典的派生结论。** 各链路依据自身字段决定是否执行，不交叉判定：

```
text2sql: subjects 或 metrics 非空？ ─┬─ 是 → 检查槽位
                                     │       ├─ 齐全 → 执行 Text2SQL
                                     │       └─ 缺失 → 追问补槽位
                                     │
                                     └─ 否 → 不跑 Text2SQL

rag: need_composition / need_policy_basis / need_caliber_explanation 任一为 true？
                                     ─┬─ 是 → 执行 RAG
                                     └─ 否 → 不跑 RAG
```

五种调度结果由两个链路的执行决策组合而成：

| Text2SQL | RAG | 调度结果 |
|----------|-----|---------|
| 执行 | 不执行 | **纯 Text2SQL** |
| 不执行 | 执行 | **纯 RAG** |
| 执行 | 执行 | **混合（双链路并行）** |
| 槽位缺失 | 执行/不执行 | **追问补槽位** |
| 不执行 | 不执行 | **闲聊兜底** |

| 字段 | 类型 | 示例值 | 旧框架可复用来源 |

### 3.4 rag — 检索增强专用于集（5个）

统一意图提取层只负责回答"用户需要 RAG 做什么"，不负责"RAG 该怎么检索"。
`search_query` 的拼接、关键词扩展等检索策略由 RAG Agent 内部自行处理，
Agent 拿着 `original_question` + `need_*` 布尔值自行改写和检索，
与旧 dataQuery 中 Agent 的行为一致。

| 字段 | 类型 | 示例值 | 说明 |
|------|------|--------|------|
| `need_policy_basis` | `bool` | `false` | 是否需检索政策/文件/规定/措施依据 |
| `need_caliber_explanation` | `bool` | `false` | 是否需检索口径解释/定义/怎么算 |
| `need_composition` | `bool` | `true` | 是否需检索构成/分类/包括哪些 |
| `need_data_value` | `bool` | `false` | 检索时是否附加数据类关键词（"收入金额/总计/数值"），**仅影响 RAG 内部检索策略，不参与路由** |
| `original_question` | `str` | 用户原始问题原文 | RAG Agent 自行改写后检索 |

## 6. 提取流程设计

```
question + history_messages + region_code
                    │
                    ▼
    ┌───────────────────────────────┐
    │  第一阶段：LLM 结构化提取       │
    │  (复用 INTENT_PROMPT 模板，     │
    │   扩充 rag 侧 4 个 need_* 字段) │
    └───────────────┬───────────────┘
                    │
              ┌─────┴─────┐
              │ 成功？      │
              └─────┬─────┘
           是       否
            │        │
            ▼        ▼
    ┌──────────┐  ┌──────────────────┐
    │ 规则补全  │  │ 全量规则兜底       │
    │ 逐字段    │  │ fallback_query    │
    │ 校验修正  │  │ _plan()           │
    └────┬─────┘  └────────┬─────────┘
         │                 │
         └────────┬────────┘
                  │
                  ▼
    ┌───────────────────────────────┐
    │  第二阶段：拆分投影             │
    │  text2sql ← 结构化、精确化      │
    │  rag     ← 扩展化、检索优化     │

                    ▼
```

`dispatch_execution.py` 中的调度逻辑：

```python
def dispatch(intent_dict: dict) -> DispatchPlan:
    t2s = intent_dict["text2sql"]
    rag = intent_dict["rag"]

    run_t2s = bool(t2s["subjects"] or t2s["metrics"])
    run_rag = rag["need_composition"] or rag["need_policy_basis"] or rag["need_caliber_explanation"]

    if run_t2s and not _slots_ready(t2s):
        return DispatchPlan(clarify=True, missing=_missing_slots(t2s))

    return DispatchPlan(
        text2sql=run_t2s,
        rag=run_rag,
    )
```
